# Лабораторная работа: умножение матриц на CPU и GPU (CUDA)

## Что реализовано

Проект содержит две функции умножения матриц:

1. `multiply_cpu(...)` — вычисление на CPU.
2. `multiply_gpu(...)` — вычисление на GPU с использованием CUDA.

Программа:

- генерирует две случайные матрицы размера `M x K` и `K x N`;
- умножает их на CPU и GPU;
- сравнивает результаты;
- выводит время вычисления;
- умеет запускать серию экспериментов для нескольких размеров и сохранять результаты в CSV.

## Описание реализации

### 1. Умножение на CPU

CPU-версия реализована в функции `multiply_cpu(...)`.

Используется классическая схема умножения матриц:

C[i][j] = sum_{k=0}^{K-1} A[i][k] * B[k][j]

где:

* A — матрица размера M × K
* B — матрица размера K × N
* C — результирующая матрица размера M × N
* i = 0, …, M-1
* j = 0, …, N-1
* k = 0, …, K-1

Чтобы улучшить локальность данных в памяти, перед основным циклом вторая матрица транспонируется. После этого при вычислении элемента результата доступ к элементам идет по непрерывным участкам памяти, что уменьшает количество cache misses и ускоряет CPU-версию по сравнению с полностью наивной реализацией.

### 2. Умножение на GPU

GPU-версия реализована в функции `multiply_gpu(...)` и CUDA kernel `tiled_matmul_kernel`.

Используется блочное умножение (tiling) с применением `shared memory`.

Каждый блок потоков CUDA вычисляет один тайл результирующей матрицы `C`.
Каждый поток внутри блока отвечает за вычисление одного элемента `C[row][col]`.

Что именно распараллелено:

- вычисление элементов результирующей матрицы распараллелено по потокам GPU;
- разные блоки независимо обрабатывают разные фрагменты матрицы результата;
- внутри каждого блока потоки совместно загружают тайлы матриц `A` и `B` в `shared memory`.

Почему это эффективно:

- элемент `C[row][col]` можно вычислять независимо от других элементов;
- GPU хорошо подходит для большого числа одинаковых арифметических операций;
- использование `shared memory` уменьшает количество обращений к глобальной памяти;
- один и тот же загруженный тайл повторно используется многими потоками блока.

### 3. Проверка корректности

Корректность GPU-вычисления проверяется сравнением с CPU-результатом.

Для сравнения рассчитываются:

- `max_abs_error` — максимальная абсолютная ошибка;
- `max_rel_error` — максимальная относительная ошибка;
- флаг `correct`, показывающий, уложился ли результат в допуски.

По умолчанию используются допуски:

- абсолютный: `1e-3`
- относительный: `1e-3`

### 4. Измерение времени

Измеряются:

- `cpu_time_ms` — полное время умножения на CPU;
- `gpu_total_time_ms` — полное время GPU-вычисления, включая копирование данных и kernel;
- `gpu_kernel_time_ms` — время только CUDA kernel;
- `speedup = cpu_time_ms / gpu_total_time_ms`.

В отчете рекомендуется использовать именно `gpu_total_time_ms`, потому что это честное end-to-end сравнение с CPU.

## Сборка

### С CUDA

На машине с установленным CUDA Toolkit:

```bash
cmake -S . -B build
cmake --build build
```

Если `nvcc` найден, CMake автоматически соберет CUDA-версию.

### Без CUDA

Если CUDA-компилятор отсутствует, проект все равно соберется, но GPU-часть будет заменена заглушкой. Это полезно для проверки CPU-части и CLI.

## Запуск

### Один эксперимент

```bash
./build/matmul --m 1024 --k 1024 --n 1024 --tile 16
```

Пример выводимой информации:

- размеры матриц;
- время CPU;
- время GPU;
- ускорение;
- корректность результата.

### Серия экспериментов

```bash
./build/matmul --benchmark --sizes 100,250,500,1000,1500,2000 --csv benchmark_results.csv
```

Программа выведет таблицу на экран и сохранит CSV-файл.

## Построение графика

После получения `benchmark_results.csv` можно построить график:

```bash
python3 scripts/plot_speedup.py benchmark_results.csv speedup.png
```

Скрипт строит:

- график времени CPU и GPU;
- график ускорения GPU относительно CPU.

## Краткий вывод для отчета

Основная идея распараллеливания состоит в том, что элементы результирующей матрицы можно вычислять независимо. В CUDA-реализации каждый поток считает один элемент результата, а блоки потоков совместно используют тайлы исходных матриц через `shared memory`. За счет этого GPU лучше масштабируется на больших размерах матриц и обычно дает заметное ускорение по сравнению с CPU.
