# Лабораторная работа: сумма элементов вектора на CPU и GPU

## Описание задачи

Нужно реализовать сложение элементов вектора размером от `1 000` до `1 000 000`
элементов двумя способами:

1. Последовательно на CPU.
2. Параллельно на GPU с использованием CUDA.

Программа должна выводить:

- сумму элементов вектора;
- время вычисления на CPU;
- время вычисления на GPU;
- ускорение GPU относительно CPU.

Для удобства воспроизводимых экспериментов входной вектор в программе
генерируется автоматически: элементы - целые числа в диапазоне `1..100` с
фиксированным seed.

## Описание реализации

### 1. Суммирование на CPU

Функция `sumVectorCPU` проходит по всем элементам вектора обычным циклом и
накапливает сумму в переменной типа `long long`.

Особенности:

- алгоритм полностью последовательный;
- вычислительная сложность `O(n)`;
- используется как базовый вариант для сравнения с GPU.

### 2. Суммирование на GPU

Функция `sumVectorGPU` использует **параллельную редукцию** (`reduction`).

Идея:

1. Исходный вектор копируется из оперативной памяти на GPU.
2. CUDA-ядро запускается в виде сетки блоков.
3. Каждый поток обрабатывает независимые элементы массива и считает локальную
   частичную сумму.
4. Внутри блока частичные суммы складываются в **shared memory**, что быстрее,
   чем постоянно обращаться к global memory.
5. Каждый блок записывает один промежуточный результат.
6. Если промежуточных результатов больше одного, запускается следующий шаг
   редукции, пока не останется одно число - итоговая сумма.

### Что именно было распараллелено и почему

На GPU распараллелены **операции сложения независимых элементов вектора**.
Это возможно, потому что на первом этапе сумма разных частей массива может
считаться одновременно разными потоками без конфликтов.

Почему выбран именно такой подход:

- сложение элементов вектора хорошо разбивается на независимые подзадачи;
- reduction позволяет уменьшать количество данных поэтапно;
- использование `shared memory` внутри блока снижает число обращений к
  медленной глобальной памяти;
- GPU особенно эффективен на больших массивах, где есть достаточно параллельной
  работы для большого числа потоков.

Итог: на CPU сумма считается одним потоком, а на GPU одновременно работают
сотни и тысячи потоков, каждый из которых обрабатывает свою часть входного
массива.

## Особенности измерения времени

- Для CPU измеряется среднее время одного запуска функции суммирования.
- Для GPU измеряется среднее время одного запуска, включающее:
  - копирование массива `Host -> Device`;
  - выполнение CUDA-ядер;
  - копирование результата `Device -> Host`.
- Выделение и освобождение памяти CUDA в замер не включается, чтобы сравнение
  было более корректным именно по вычислительной части алгоритма.
- Для уменьшения погрешности измерения каждый эксперимент повторяется несколько
  раз, после чего берётся среднее значение.

## Сборка

Требования:

- NVIDIA GPU;
- установленный CUDA Toolkit;
- компилятор `nvcc`;
- `CMake`;
- генератор сборки для `CMake`:
  - на Windows обычно подходит Visual Studio 2022;
  - либо можно использовать Ninja.

Сборка через CMake:

```bash
cmake -S . -B build
cmake --build build --config Release
```

## Запуск

Запуск набора экспериментов по умолчанию:

```bash
./build/vectorsum
```

Запуск с собственными размерами:

```bash
./build/vectorsum 1000 10000 100000 500000 1000000
```

Запуск с фиксированным числом повторов для каждого размера:

```bash
./build/vectorsum --repeats 100 1000 10000 100000 500000 1000000
```

На Windows в `PowerShell` команды запуска выглядят так:

```powershell
.\build\vectorsum.exe
.\build\vectorsum.exe 1000 10000 50000 100000 250000 500000 1000000
.\build\vectorsum.exe --repeats 100 1000 10000 50000 100000 250000 500000 1000000
```

## Вывод

В работе реализованы два варианта суммирования элементов вектора:

- последовательный алгоритм на CPU;
- параллельный алгоритм на GPU с использованием CUDA reduction.

Главная идея распараллеливания состоит в том, что сложение элементов массива
можно выполнять независимо для разных частей входного набора данных, а затем
объединять частичные суммы.

Ожидаемый результат экспериментов:

- на малых размерах GPU может уступать CPU из-за накладных расходов на передачу
  данных и запуск ядра;
- на больших размерах GPU обычно показывает лучшее время и даёт ускорение.
